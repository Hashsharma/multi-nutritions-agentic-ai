Multi-Turn Conversation
A multi-turn conversation is a dialogue where the interaction happens across multiple back-and-forth exchanges, and the system remembers earlier context to respond correctly.

I’d handle multi-turn conversations using a layered memory architecture: recent chat history for immediate context, summarized history for scalability, and retrieval-based long-term memory using embeddings and a vector database. Before each response, I’d retrieve only the most relevant memories and combine them with recent turns to stay within token limits. I’d also classify memories by importance, avoid storing sensitive data, and periodically summarize old conversations to reduce cost and latency while maintaining coherence.


I’d reduce hallucinations primarily by grounding the chatbot using Retrieval-Augmented Generation, where responses are generated from trusted retrieved documents instead of only model memory. I’d also use strong system prompts, confidence thresholds, structured outputs, and validation layers to prevent fabricated responses. For high-risk domains, I’d add human review and external verification systems.

1️⃣ What semaphore can do
Controls how many requests run concurrently per LLM.
If a request arrives and semaphore slots are full:
The request waits until a slot frees up.
Semaphore does not queue requests persistently; it just blocks them in memory.

✅ Pros: Simple, ensures concurrency limit.
❌ Cons: Requests pile up in memory → high latency if too many arrive at once.

4️⃣ Recommended pattern

For production-level LLM usage:

Semaphore per LLM worker → controls concurrency
Queue system → stores pending requests safely
Timeout + fallback + retry → ensures reliability
Semaphore: “how many requests can run now”
Queue: “what requests are waiting to be processed”
Timeout/fallback: “what to do if a request fails or takes too long”


. Assigning a Worker to a Core
When you assign a worker to a specific core (CPU affinity), it hints to the OS: “run this worker only on this core.”
The OS then tries to schedule all the worker’s threads on that core.
Effect: That core becomes mostly dedicated to that worker’s tasks, reducing context switching and improving performance.

✅ So yes, in that sense, assigning a worker to a core effectively reserves it, but:

Other OS tasks can still run on that core if needed.
It’s not “exclusive” unless you specifically isolate the core at the OS level (e.g., using isolcpus in Linux).

Boilerplate text: Standardized text that appears across multiple documents, such as disclaimers, headers, footers, copyright notices, terms and conditions, or templated language.

To handle irrelevant or noisy documents, I use a combination of preprocessing, better retrieval, and post-retrieval filtering.

First, during ingestion, I clean and structure the data by removing boilerplate content, duplicates, and low-quality text, and I use intelligent chunking to preserve semantic coherence.

Second, during retrieval, I rely on a hybrid approach—combining keyword-based search like BM25 with dense retrieval using bi-encoder embeddings. This helps balance exact match and semantic match, reducing irrelevant hits.

Third, I apply re-ranking using a cross-encoder, which scores query–document pairs jointly and filters out weak matches from the top-k results.

Finally, I often set similarity thresholds or top-k limits and may add metadata filters (like source type or domain) to further reduce noise before passing context to the generation model.

To evaluate the performance of individual agents, 
I measure them in isolation as well as within the full pipeline using task-specific metrics.

First, I define clear responsibilities for each agent (e.g., retriever, planner, generator, validator), and assign metrics aligned to their role. For example, a retriever is evaluated using Recall@k or Precision@k, while a generator is evaluated on answer correctness and faithfulness.

Second, I create benchmark datasets or test cases where each agent can be evaluated independently. This helps isolate whether errors are coming from a specific agent or from interactions between agents.

Third, I log intermediate outputs at every stage of the pipeline. This allows me to trace failures and understand which agent contributed to an incorrect final result.

Example metrics per agent
Retriever → Precision@k, Recall@k
Generator → correctness, faithfulness
Planner → task completion accuracy
Validator → error detection rate


I evaluate individual agents using role-specific metrics, isolate them with benchmark tests, log intermediate outputs for debugging, and validate their impact through A/B testing within the full pipeline

CRAG (Corrective Retrieval-Augmented Generation) is an improved version of standard RAG where the system does not blindly trust retrieved documents. Instead, it checks the quality of retrieval results and corrects them if they are weak or irrelevant before generating an answer.

CRAG = RAG + “self-check + auto-fix retrieval”

BERT is generally faster and better for text understanding tasks because it uses bidirectional attention and processes the whole input at once. GPT is better for text generation because it is autoregressive and predicts tokens sequentially. Modern AI systems often use encoder models for retrieval and GPT-style decoder models for generation.


Self-attention is the Transformer mechanism that computes relationships between tokens using Q, K, and V matrices. Standard attention materializes the full attention matrix, which requires O(N²) memory. FlashAttention is a memory-efficient implementation that computes the same attention result in blocks without storing the full attention matrix. It reduces memory usage, improves GPU utilization, and enables faster training and inference for long-context LLMs while producing mathematically equivalent outputs.
